# 创业板股票实时监控程序需求计划书

## 1. 项目背景

中国 A 股创业板股票波动较大，人工逐只盯盘成本高。本项目希望构建一个实时监控程序，自动获取创业板股票每分钟 K 线数据，并在个股出现显著异动时通知用户，辅助盘中观察和决策。

本项目仅做行情监控与告警提醒，不自动交易，不直接给出买卖建议。

## 2. 项目目标

### 2.1 核心目标

- 实时监控创业板全部股票。
- 按分钟生成或读取每只股票的 1 分钟 K 线。
- 检测异常波动，包括快速上涨、快速下跌、成交量放大、振幅扩大等。
- 在触发规则后，通过指定渠道通知用户。
- 保存监控记录，便于复盘。

### 2.2 非目标

- 不做自动下单。
- 不承诺行情源达到交易所级低延迟。
- 初期不做复杂机器学习预测。
- 初期不覆盖沪深主板、科创板、北交所，后续可扩展。

## 3. 股票范围

### 3.1 监控标的

创业板股票通常为深圳证券交易所创业板上市股票，股票代码主要以 `300`、`301` 开头。

### 3.2 股票列表维护

需要支持以下方式之一：

- 从行情数据源拉取创业板股票列表。
- 从公开接口定期同步股票基础信息。
- 本地维护股票池文件，定期更新。

建议第一阶段采用“自动同步 + 本地缓存”的方式：

- 每个交易日前同步一次股票列表。
- 盘中使用本地缓存，避免频繁请求基础数据接口。
- 支持手动排除停牌、退市整理、长期无成交股票。

## 4. 行情数据需求

### 4.1 必需字段

每只股票每分钟至少需要以下数据：

- 股票代码
- 股票名称
- 分钟时间戳
- 开盘价
- 最高价
- 最低价
- 收盘价
- 成交量
- 成交额
- 昨收价或前一分钟收盘价

### 4.2 数据频率

- 盘中每 3 至 10 秒拉取一次实时行情快照。
- 程序内部按自然分钟聚合为 1 分钟 K 线。
- 如果数据源直接提供 1 分钟 K 线，可优先读取源端 K 线，再补充快照数据。

### 4.3 交易时间

A 股常规交易时间：

- 上午：09:30 - 11:30
- 下午：13:00 - 15:00

集合竞价阶段可作为后续扩展：

- 09:15 - 09:25
- 14:57 - 15:00 深市收盘集合竞价

第一阶段建议只监控连续竞价时段，降低异常数据处理复杂度。

## 5. 异动判定规则

“巨大波动”需要可配置，避免固定阈值无法适配不同市值、价格和活跃度的股票。

### 5.1 价格异动

建议第一阶段支持以下规则：

- 单分钟涨跌幅超过阈值，例如 `>= 1.5%`。
- 最近 3 分钟累计涨跌幅超过阈值，例如 `>= 3%`。
- 分钟内振幅超过阈值，例如 `(最高价 - 最低价) / 前一分钟收盘价 >= 2%`。
- 股价快速接近涨跌停，例如距离涨停价或跌停价小于 `1%`。

### 5.2 成交量异动

建议支持以下规则：

- 当前分钟成交量超过过去 N 分钟均量的倍数，例如 `>= 3 倍`。
- 当前分钟成交额超过固定阈值，例如 `>= 1000 万元`。
- 价格上涨同时成交量放大。
- 价格下跌同时成交量放大。

### 5.3 综合异动评分

第二阶段可以引入综合评分：

```text
异动分 = 价格波动分 + 成交量放大分 + 振幅分 + 趋势连续性分
```

当分数超过阈值时告警。这样可以减少单一指标造成的误报。

### 5.4 告警去重

需要避免同一只股票在短时间内重复刷屏：

- 同一股票同一规则触发后，设置冷却时间，例如 3 分钟。
- 更高级别异动可以突破冷却限制。
- 每条告警记录触发原因和触发数据。

## 6. 通知需求

### 6.1 第一阶段通知方式

建议优先支持：

- 控制台输出。
- 本地日志文件。
- 企业微信机器人或钉钉机器人。

### 6.2 后续通知方式

可扩展：

- Telegram Bot
- 邮件
- 手机短信
- 桌面通知
- Web 页面实时推送

### 6.3 告警内容

告警消息至少包含：

- 股票代码和名称
- 当前时间
- 当前价格
- 1 分钟涨跌幅
- 3 分钟涨跌幅
- 当前分钟成交额
- 触发规则
- 简短解释

示例：

```text
[创业板异动] 300XXX 某某股份
时间：10:24
现价：18.52
1分钟涨幅：1.8%
3分钟涨幅：3.4%
成交额：2560万元
触发：单分钟涨幅超过阈值 + 成交量放大
```

## 7. 数据源方案

### 7.1 可选数据源

需要进一步确认可用性、稳定性、费用和授权边界。候选包括：

- 富途 OpenAPI：实时性较好，但需要本地 OpenD，并受账户权限限制。
- Tushare Pro：适合基础数据和部分行情数据，实时能力和权限取决于积分与接口。
- AkShare：适合快速原型，数据源多为公开网页接口，稳定性可能波动。
- 东方财富、新浪等公开行情接口：易接入，但接口稳定性和合规性需要评估。
- 券商或商业行情 API：稳定性更好，通常需要付费。

### 7.2 初步建议

当前已决定直接使用富途 OpenAPI 作为主数据源：

- 优点：实时行情能力较好，Python SDK 成熟。
- 风险：A 股实时行情权限、订阅限制、连接稳定性、OpenD 配置。
- 当前本机实测：深市股票基础信息可拉取，过滤出创业板股票 `1408` 只；富途 `K_1M` 订阅额度显示总量 `1000`，因此不能一次订阅全部创业板分钟 K 线。

第一阶段不接入 AkShare 或 Tushare。由于富途全量创业板订阅受限，短期策略是从创业板股票中固定随机抽取 `1000` 只监控，其余约 `408` 只暂不监控；后续如要覆盖全部创业板，需要分批轮询、升级行情权限，或引入备用数据源。

更详细的数据源评估见：[实时股票数据源调研](data-sources.md)。

## 8. 初步技术架构

### 8.1 模块划分

```text
行情源适配器
  -> 股票列表同步

  -> 异动检测引擎
```

## 12. 关键风险

### 12.1 行情实时性与权限

不同数据源对 A 股实时行情的权限不同。有些接口可能是延迟行情，或者盘中频率不足。需要在开发前实测。

### 12.2 接口稳定性

公开免费接口可能变更、限流或返回异常数据。需要设计重试、降级和异常保护。

### 12.3 全量股票性能

创业板股票数量较多，如果每几秒全量请求一次，需要注意接口限流、网络延迟和处理性能。

### 12.4 误报与漏报

固定阈值容易误报，需要保留配置能力，并通过历史回放不断调整。

### 12.5 合规边界

程序只做个人行情监控和提醒。若要公开提供服务、转发实时行情或商业化，需要确认行情授权和相关合规要求。
